MediAI Platform

An AI-powered intelligent digital care platform combining symptom triage, doctor discovery, appointment scheduling, medical report analysis, emergency routing, and patient engagement — built as a full-stack demo with multi-agent AI orchestration.

React 18 + TypeScript FastAPI + Python 3.12 PostgreSQL + pgvector Redis MinIO (S3) Groq / Gemini LLM Docker Compose

System Architecture

MediAI is a Dockerized monorepo with five services orchestrated by docker-compose.yml. The React SPA talks to a FastAPI backend over REST. The backend persists data in PostgreSQL, caches session state in Redis, stores files in MinIO, and calls external LLM APIs for AI features.

Startup & Build Flow

The project is designed for one-command startup. No manual DB setup required.

Prerequisites

  • Docker Desktop installed
  • Copy .env.example → .env
  • Set at least one LLM API key: GROQ_API_KEY or GEMINI_API_KEY

Start Commands

# Windows
.\start.ps1

# Mac / Linux
docker compose up --build

# Full reset (wipes database)
docker compose down -v

What Happens on First Boot

  1. Docker builds images Backend: Python 3.12-slim + pip install. Frontend: Node 20 + npm install.
  2. Services start with health checks PostgreSQL, Redis, and MinIO must be healthy before the API starts.
  3. entrypoint.py runs in the API container Waits for DB → runs Alembic migrations → seeds demo data → starts Uvicorn with hot reload.
  4. Frontend dev server starts Vite on port 5173, proxying API calls to the backend container. 
# entrypoint.py sequence
wait_for_db()
alembic upgrade head          # Apply DB migrations
python seed.py                # Users, doctors, patients
python seed_demo.py           # Extra slots & medications
uvicorn app.main:app --reload # Start API on :8000

Multi-Agent AI Architecture

The core intelligence layer. A Supervisor routes each patient message to a Specialist Agent, which uses an LLM plus scoped database tools to take real actions (book appointments, assess symptoms, analyze reports).

Each specialist returns structured JSON: reply, optional tool call, session_patch, and handoff_to another agent. Only one tool call per turn is allowed — tools read/write real PostgreSQL data, not hallucinated responses.

Key User Flows

1. Guest Chat → Signup → Patient

sequenceDiagram
    participant U as User (Browser)
    participant G as GuestChatWidget
    participant API as /api/v1/guest
    participant Redis as Redis Session
    participant DB as PostgreSQL

    U->>G: Open landing page chat
    G->>API: POST /guest/session
    API->>Redis: Create guest session
    U->>G: "I have a headache"
    G->>API: POST /guest/message
    API->>API: chat_orchestrator → Supervisor → Triage
    API-->>G: AI reply + quick actions
    U->>G: Enter email + OTP
    G->>API: POST /guest/register
    API->>DB: Create User + Patient
    API->>Redis: Migrate guest session → patient
    API-->>G: JWT tokens

2. Patient Chat Message Flow

  1. Frontend sends message POST /api/v1/chat/conversations/{id}/messages
  2. Message saved to PostgreSQL Conversation history loaded; patient context assembled from profile, meds, allergies.
  3. Emergency pre-check emergency_detection.py scans for crisis keywords before AI routing.
  4. Supervisor picks specialist LLM routing based on message intent, active agent, and care goal.
  5. Specialist processes + optional tool call e.g. assess_symptoms, book_slot, analyze_report
  6. Reply + UI hints returned Booking buttons, yes/no prompts, slot pickers rendered by frontend components.

3. Appointment Booking

Can happen via chat (Scheduling Agent) or the Doctors page. Backend enforces slot availability, conflict prevention, and sends HTML email notifications to both patient and doctor.

4. Report Upload & Analysis

Patient uploads PDF/DOCX/XLSX → stored in MinIO → parsed with pdfplumber/python-docx/openpyxl → LLM analyzes content → results saved and surfaced in chat and dashboard vitals.

Security & Cross-Cutting Concerns

Authentication

JWT access tokens (30 min) + refresh tokens (7 days). OTP email verification for signup.

Authorization (RBAC)

Three roles: patient, doctor, admin — enforced in API dependencies and frontend routes.

Rate Limiting

Configurable limits on chat (60/min) and auth (10/min) endpoints via Redis.

Audit Logging

Middleware logs sensitive actions to AuditLog table with request IDs.

PHI Protection

Sensitive fields (content, OCR text, passwords) masked in application logs.

Healthcare Policy

Clinical disclaimers, off-topic filtering, responsible AI guidance rules.

Demo / Educational Use Only. Not intended for production medical use without proper HIPAA/GDPR compliance review, security hardening, and clinical validation.
