Start Here — New Developer Guide

Read this first. ~5 minutes to orient yourself.

What is MediAI? A full-stack healthcare demo: patients chat with a multi-agent AI assistant for triage, booking, and report questions; doctors manage consultations in a portal; admins oversee users.

Step 1

Clone & configure

Copy .env.example → .env. Add a Groq or Gemini API key.

Step 2

Start Docker

Run docker compose up --build or .\start.ps1 on Windows.

Step 3

Explore the app

Open localhost:5173. Log in as patient, doctor, or admin (see Quick Start).

Multi-Agent

This project does use a multi-agent style architecture:

  • Supervisor routes patient messages to the right specialist
  • Specialist agents: triage, scheduling, education, report, refill, safety, follow-up
  • Tool execution for booking, reports, refills, and related actions
  • Flow state management across conversation turns
  • Emergency / urgent consult detection and guest → patient flows

This is more than a simple single-prompt chatbot, but not the most advanced multi-agent system (no deep agent hierarchies, no LangGraph orchestration in active use, no heavy autonomous planning loops).

RAG (Retrieval-Augmented Generation)

This project does not implement complex modern RAG as described in advanced architecture articles. What exists is closer to simple / lightweight retrieval and context use.

Comparison

Complex RAG (typical advanced stack) This project
Vector embeddings + semantic search pgvector is in DB setup, but no full embedding/retrieval pipeline in chat
Chunking + indexing + hybrid search Reports are parsed as text and analyzed by LLM directly
Query rewriting / reranking / multi-hop RAG Not implemented
Agentic RAG loops Not implemented

What This Project Actually Uses

  • Direct database context (patient profile, medications, allergies, appointments)
  • Report upload → OCR/text extraction → LLM analysis (not vector RAG)
  • Rule-based symptom and emergency detection
  • Conversation memory stored as facts
  • LLM + tools + supervisor routing

Bottom Line

This is a solid, feature-rich medical chatbot with multi-agent orchestration, but it is not a “complex RAG + advanced multi-agent” architecture like advanced Medium-style articles describe.

It is practical mid-level:

  • Smart routing
  • Tools
  • Context injection

It is not a full enterprise RAG stack with vector retrieval, hybrid search, reranking, or agentic retrieval loops.

Upgrade Vision — Complex Multi-Agent + RAG

Goal: evolve MediAI from mid-level orchestration to an enterprise-grade clinical AI platform with retrieval-grounded answers, auditable agent workflows, and safer medical decision support.

Recommended strategy: upgrade in layers — RAG foundation first, then agent orchestration, then agentic loops. Do not jump straight to “full complex” in one release.

Layer Today Target (complex)
Knowledge DB rows + report JSON Indexed medical knowledge + patient-specific vector stores
Retrieval Load full patient context Hybrid RAG (semantic + keyword + metadata filters)
Reasoning Supervisor + specialist LLM calls LangGraph state machine + planner/reviewer agents
Tools Booking, reports, refill tools Tools + retrieval tools + clinical validators
Safety Rule-based emergency detection Rules + LLM safety classifier + citation-required answers

Suggested RAG Upgrades (Simple → Complex)

Phase R1 — Foundation (Naive → Indexed RAG)

Priority 1 Vector store on existing pgvector

  • Chunk & embed uploaded reports (lab PDFs, discharge summaries) after OCR — store chunks in a document_chunks table with pgvector embeddings.
  • Embed patient artifacts: SOAP notes, triage summaries, conversation memory facts, medication lists.
  • Retrieval at chat time: top-k semantic search scoped by patient_id + document type.
  • Replace “inject entire report analysis JSON” with “retrieve only relevant chunks”.

Files to extend later: report_service.py, new rag/embedding_service.py, rag/retrieval_service.py, Alembic migration for chunk tables.

Phase R2 — Advanced RAG (Indexed → Hybrid + Reranked)

Priority 2 Hybrid retrieval + reranking

  • Hybrid search: combine pgvector cosine similarity + PostgreSQL full-text (BM25/tsvector) for lab test names, drug names, ICD codes.
  • Metadata filters: date range, document type (lab vs prescription), doctor_id, appointment_id.
  • Cross-encoder reranker (e.g. Cohere rerank, bge-reranker, or LLM rerank) on top-20 → top-5 chunks.
  • Query rewriting: expand patient question into clinical search queries (“headache 3 days” → “cephalgia duration triage fever”).

Phase R3 — Agentic RAG (Complex)

Priority 3 Multi-step retrieval loops

  • Retriever agent decides: search patient history, search medical guidelines, or both.
  • Multi-hop RAG: initial retrieval → gap analysis → follow-up retrieval (e.g. find lab → find related meds → find allergies).
  • Self-RAG / CRAG pattern: model checks if retrieved context is sufficient; if not, reformulate query and retrieve again.
  • Citations required: every clinical claim links to chunk IDs (report page, note date) shown in doctor portal.

Phase R4 — Knowledge bases beyond patient data

Priority 4 External medical knowledge RAG

  • Ingest curated sources: clinical guidelines, drug interaction DBs, hospital protocols (not open web scrape for production).
  • Separate index namespace: global_clinical_kb vs patient_private — strict RBAC at retrieval time.
  • Version and audit knowledge snapshots for compliance.

Suggested Multi-Agent Upgrades (Medium → Complex)

Phase A1 — Orchestration hardening

Priority 1 LangGraph state machine (use existing dependency)

  • Replace ad-hoc flow_state + supervisor if/else with explicit graph nodes: route → retrieve → reason → tool → validate → respond.
  • Persist graph state per conversation (checkpointing) for resume after disconnect.
  • Each specialist becomes a subgraph (triage subgraph, scheduling subgraph).
  • Extend: multi_agent/supervisor.py, new multi_agent/graph.py.

Phase A2 — Planner + specialist + critic pattern

Priority 2 Hierarchical agents

  • Planner agent: breaks patient goal into steps (triage → risk assess → recommend action → offer booking).
  • Specialist workers: current agents (triage, scheduling, report) execute steps.
  • Critic / safety agent: reviews draft reply for hallucination, prescription requests, missing disclaimers, emergency miss.
  • Human-in-the-loop node for high-risk outputs before showing to patient.

Phase A3 — New clinical agents

Priority 2 Additional specialists to add

  • Clinical retrieval agent — owns all RAG tool calls (not embedded in every specialist).
  • Medication safety agent — cross-checks meds vs allergies, interactions (retrieval + rules).
  • Documentation agent — drafts SOAP, patient summary, doctor handoff from chat + retrieval context.
  • Follow-up orchestrator — proactive post-visit check-ins using appointment + triage history.

Phase A4 — Agentic tool ecosystem

Priority 3 Rich tool layer

  • Standardize tools with JSON schema + idempotent execution + audit log (agent_tool_runs table).
  • New tools: search_patient_records, search_clinical_guidelines, get_drug_interactions, summarize_visit_with_citations.
  • Parallel tool calls where safe (retrieve labs + meds + allergies simultaneously).
  • Extend: services/agent_tools.py, multi_agent/specialists.py.

Phase A5 — Observability & evaluation

Priority 3 Production agent ops

  • Trace every turn: router decision, retrieved chunks, tools called, critic verdict, latency, token cost.
  • Eval suite: triage accuracy, emergency recall, RAG faithfulness, booking success rate.
  • A/B routing: compare simple vs complex path per conversation type.

Risks, Guardrails & Healthcare Constraints

Before implementing complex RAG + agents in a medical product, plan for:

  • HIPAA / privacy: embeddings of PHI must stay in your VPC; no sending patient chunks to unapproved embedding APIs without BAA.
  • Hallucination: complex RAG increases citation surface but does not eliminate wrong clinical advice — critic agent + mandatory disclaimers remain required.
  • Latency: multi-hop RAG + multiple agents can push responses from ~2s to 8–15s — use streaming and parallel retrieval.
  • Cost: embedding storage, rerankers, and extra LLM calls per turn — budget per conversation.
  • Stale index: medication or allergy changes must invalidate/update embeddings quickly.
  • Regulatory: document that AI assists triage; does not diagnose or prescribe — especially if adding clinical KB.
Recommendation: keep rule-based emergency detection (emergency_detection.py) as a hard gate before any LLM router — do not replace it with RAG or agent routing alone.

Advanced GenAI Concepts to Add (Beyond RAG + Agents)

High-impact GenAI patterns for a complex, production-grade medical assistant:

Core UX Streaming & real-time

  • SSE / WebSocket streaming for token-by-token chat — reduces perceived latency during multi-agent turns.
  • Progress events: “Searching your records…”, “Checking triage…”, “Finding slots…” while agents run.
  • Voice pipeline: AWS Transcribe / Polly or Whisper + TTS for hands-free triage (listed as readiness, not built).

Quality Structured outputs & validation

  • JSON schema / tool contracts for triage, booking, SOAP drafts — validate before DB write.
  • Pydantic + instructor pattern: force typed clinical fields (symptoms[], risk, specialty).
  • Constrained decoding for enums (risk level, appointment status) to reduce parse errors.

Efficiency Caching & context management

  • Semantic cache (Redis + embeddings): similar patient questions → cached safe answers.
  • Prompt caching (Bedrock / provider-native): reuse system + patient profile tokens.
  • Conversation summarization: compress long chats into rolling summary + episodic memory index.
  • Context budget router: decide what fits in window (RAG chunks vs full history).

Safety Guardrails & evaluation

  • LLM safety classifier layer after generation (toxicity, self-harm, prescription requests).
  • Amazon Bedrock Guardrails or NeMo Guardrails for topic + PII redaction.
  • RAGAS / DeepEval offline eval: faithfulness, context recall, emergency detection recall.
  • LLM-as-judge for triage quality on golden datasets before each release.
  • Human feedback loop: doctor thumbs-up/down on AI summaries → fine-tune prompts or rankers.

Intelligence Model routing & ensembles

  • Router model: cheap model for greetings; large model for triage; vision model for images.
  • Fallback chain: Bedrock Claude → Groq → Gemini → offline rules (already partial).
  • Specialist fine-tunes (optional): small model for symptom extraction; keep general LLM for dialogue.
  • Uncertainty scoring: when confidence low → ask clarifying question or escalate to human.

Multimodal Vision & documents

  • Layout-aware parsing: Textract / unstructured.io for lab tables (better than plain OCR).
  • Medical VLM pipeline: rash/wound images → structured findings + “see doctor if…” (not diagnosis).
  • Chart grounding: link vision findings to retrieved patient history via RAG.

Platform Agent protocols & memory

  • MCP (Model Context Protocol) for standardized tools (calendar, EHR, pharmacy lookup).
  • Long-term memory tiers: working memory (session) + episodic (visits) + semantic (facts index).
  • Proactive agents: EventBridge triggers post-visit follow-up, refill reminders, abnormal lab alerts.
  • Digital twin patient context graph (optional advanced): symptoms ↔ meds ↔ labs relationships.

Suggested AWS implementation phases

  1. AWS Phase 1: Terraform/CDK — VPC, RDS, ElastiCache, S3, ECR, ECS Fargate (lift-and-shift API). Foundation
  2. AWS Phase 2: CloudFront + ACM + Route 53, Secrets Manager, SES, WAF on ALB.
  3. AWS Phase 3: SQS workers for report processing, embedding jobs, email queue.
  4. AWS Phase 4: Bedrock for LLM + embeddings + Guardrails; retire external API keys in prod.
  5. AWS Phase 5: Multi-AZ, read replicas, backup/DR runbook, CloudWatch dashboards + PagerDuty alerts.
Cost note: AWS is not free for production. See full cost breakdown and low-cost complex architecture for budget-friendly options.

Cost optimization tactics for AI

  • Model routing: small/fast model for greetings & booking; large model only for triage.
  • Cache: Redis semantic cache for FAQ-style health education questions.
  • Shorter prompts: RAG top-3 chunks only — not full patient history every turn.
  • Batch embeddings: embed reports on upload (background), not during chat.
  • Rules first: keep emergency_detection.py — avoids LLM call on clear emergencies.
  • Token budgets: cap history to last N messages + summary blob.
  • Dev/staging: stay on local Docker ($0); only prod VPS runs 24/7.

When to upgrade from low-cost → AWS

  1. Need signed AWS BAA and formal HIPAA for real patient PHI at scale
  2. More than ~500–1000 daily active users or need auto-scaling
  3. Require 99.9% SLA, Multi-AZ, dedicated security team integrations
  4. Hospital asks for VPC isolation, private Bedrock, HealthLake/FHIR

DevOps, Security & Production Operations

CI/CD & release engineering

  • GitHub Actions: lint → pytest → Docker build → push ECR → deploy ECS
  • Environments: dev · staging · prod with separate RDS/S3
  • Database migrations via Alembic in deploy pipeline (with rollback plan)
  • Feature flags (LaunchDarkly / AWS AppConfig) for new agents without full deploy
  • Blue/green or rolling ECS deployments — zero-downtime API updates

Security & compliance

  • Encryption: TLS in transit, KMS at rest (RDS, S3, Redis where supported)
  • IAM least-privilege roles per ECS task — no root keys in containers
  • Audit log table: who accessed which patient record, AI trace IDs
  • PHI redaction in application logs — never log raw chat or lab values
  • Penetration test + OWASP scan before go-live
  • Data retention policy + right-to-delete for patient accounts (GDPR-style)

Observability & SRE

  • LLM tracing: Langfuse, Arize Phoenix, or OpenTelemetry → CloudWatch
  • Per-conversation cost tracking (tokens, embedding calls, tool latency)
  • SLOs: chat p95 < 5s, API uptime 99.9%, emergency detection 100% recall on test set
  • Synthetic monitors: health check + sample triage conversation every 5 min
  • Error tracking: Sentry for API + frontend

Reliability & scale

  • Horizontal API scaling on CPU / request count (ECS auto-scaling)
  • Separate worker service for long jobs — never block chat request thread
  • Idempotent appointment booking (already important) + distributed locks via Redis
  • Load testing with Locust/k6 — target N concurrent chat sessions
  • RDS automated backups + tested restore procedure (RPO/RTO documented)
